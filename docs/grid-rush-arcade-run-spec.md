# Grid Rush — Arcade Run Spec (v1.0)

**Type:** change spec against the current `index.html` (v0.2 implementation at `amaldevs/grid-rush@main`). Everything not mentioned here keeps its v0.2 behavior (timer accumulation, pause/visibility, pointerdown resolution, fairness invariants, neutral penalty, WebAudio, reduced-motion, single-file constraint, `scripts/check.sh` must still pass).

## 0. Design change summary

The fixed 8-level campaign is replaced by an **arcade run**:

1. Every run starts at **wave 1**. There is no level unlocking, no level select, no "highest unlocked."
2. The game is **tougher**: one shared **time bank** drains continuously across the entire run and never resets between waves.
3. **Correct taps refund time** into the bank; clearing a wave adds a bonus. Refunds shrink as waves rise, so the economy goes net-negative and every run eventually ends. Run length = skill.
4. A run ends on **any red tap** (instant, no hearts) or when the **bank hits zero**. Score = wave reached. `bestWave` is the persistent record.

> Design note (decided, not open): the continuous bank was chosen over per-wave timers + refunds because it makes refunds load-bearing rather than cosmetic — deep waves are only survivable by playing well on the waves before them. Fallback if playtesting shows wave-to-wave bleed feels unfair: keep everything below but reset the bank to `min(bank + waveClearBonus, startBankMs)` at each wave boundary. Implement the primary model first.

## 1. Time-bank economy (normative)

```js
const ECON = {
  startBankMs: 8000,     // bank at run start
  capMs: 12000,          // bank can never exceed this (no hoarding easy waves)
  refundMs: n => Math.max(350, 650 - 15 * n),      // per correct target tap, wave n
  waveClearBonusMs: n => Math.max(500, 2000 - 100 * n), // on clearing wave n
  neutralPenaltyMs: 500  // unchanged from v0.2
};
```

- **Correct target tap:** `bank = min(bank + refundMs(n), capMs)` applied inside the input handler at the same instant the tile resolves. Show a green `+0.6s`-style float-up (mirror of the existing red `−0.5s` penalty label) and animate the bar refilling.
- **Wave clear:** apply `waveClearBonusMs(n)` (clamped to cap) at the moment the final target's `pointerdown` lands, before the transition.
- **Neutral tap:** unchanged — deduct `neutralPenaltyMs`, reset streak, immediate `RUN_OVER (timeout)` if the live bank ≤ 0.
- **Red tap:** `RUN_OVER (red_tap)` immediately. No hearts in v1. (Keep a `runHearts: 1` constant so a hearts experiment is a one-line change, but the shipped value is 1 and the hearts HUD is removed.)
- All bank mutations and end-of-run decisions use the live value (`getActiveRemaining` pattern from v0.2, renamed `getBank`), never the rAF-cached display value.
- The bar displays `bank / capMs`. Absolute thresholds, not ratios of a per-wave time: amber when bank ≤ 4000 ms, danger + pulse when bank ≤ 2000 ms.

**Economy invariant:** for every wave n, `targets(n) × refundMs(n) + waveClearBonusMs(n)` must be **less than** a competent player's realistic time cost to clear wave n once `n ≥ ~6` (net-negative regime), and **greater than or roughly equal** for `n ≤ 3` (warm-up regime where a decent player builds bank toward the cap). The constants above are the starting hypothesis; tune them from telemetry (§6), never by feel alone.

## 2. Wave generation (replaces the LEVELS array)

Waves are unbounded and generated by `waveConfig(n)`:

```js
function waveConfig(n) {
  const tiles = n <= 2 ? { cols: 3, rows: 3 }
             : n <= 4 ? { cols: 3, rows: 4 }
             : n <= 6 ? { cols: 4, rows: 4 }
             : n <= 8 ? { cols: 4, rows: 5 }
             : n <= 11 ? { cols: 5, rows: 5 }
             : n <= 14 ? { cols: 5, rows: 6 }
             : { cols: 6, rows: 6 };            // hard cap: tiles stay ≥48px
  const total = tiles.cols * tiles.rows;
  const targets = Math.min(13, 3 + Math.floor(n * 0.5));
  const reds = Math.min(total - targets - 2, 1 + Math.floor(n * 0.65));
  const neutrals = total - targets - reds;      // always ≥ 2 by the clamp above
  return { n, ...tiles, targets, reds, neutrals };
}
```

Sample values (sanity anchors, derived — do not hand-maintain a table):

| Wave | Grid | Targets | Reds | Neutrals | refund/tap | clear bonus |
|---|---|---|---|---|---|---|
| 1 | 3×3 | 3 | 1 | 5 | 635 | 1900 |
| 3 | 3×4 | 4 | 2 | 6 | 605 | 1700 |
| 5 | 4×4 | 5 | 4 | 7 | 575 | 1500 |
| 8 | 4×5 | 7 | 6 | 7 | 530 | 1200 |
| 10 | 5×5 | 8 | 7 | 10 | 500 | 1000 |
| 15 | 6×6 | 10 | 10 | 16 | 425 | 500 |
| 20 | 6×6 | 13 | 14 | 9 | 350 | 500 |

Wave 1 is intentionally harder than the old level 1 (reds present immediately, 8 s bank, not 10 s): the run model assumes replay, so there is no "first level must be ~98% clearable" requirement anymore. The tutorial job moves to the one-time intro line.

## 3. States and flow

```
BOOT → MENU → RUN_INTRO → PLAYING ⇄ PAUSED
                          PLAYING → WAVE_CLEAR → PLAYING (next wave, bank carries)
                          PLAYING → RUN_OVER → PLAYING (new run, wave 1) | MENU
```

- **MENU:** "Play" starts a run at wave 1. Shows `Best: Wave N` (and `Runs: M` if trivially available). **Delete** the level strip, `highestUnlocked`, per-level buttons, "MVP Complete" screen and all reachable code for them.
- **RUN_INTRO:** the v0.2 LEVEL_INTRO overlay, shown **once per session, only at wave 1 of the first run**: "Tap ringed faces · Reds end the run · Good taps add time". Subsequent runs start after the 250 ms reset only.
- **WAVE_CLEAR (new, replaces the WIN overlay):** entered on the final target tap. **The bank is frozen for the entire state** (fairness: no drain while the player cannot act). Sequence, total ≤ 600 ms: apply clear bonus with a visible bar-fill + `+N.Ns` label → flash "Wave {n+1}" centered (no modal card, no buttons) → rebuild grid from `waveConfig(n+1)` → unfreeze, resume PLAYING. rAF loop is cancelled on entry and restarted with a fresh `lastTick` on exit (single-loop guarantee carries over).
- **RUN_OVER (replaces FAIL):** overlay shows reason ("You tapped a Red!" / "Out of time!"), `Wave N reached`, and `Best: Wave M` — with a "New best!" state when improved. Actions: **Run again** (primary, one tap, < 1 s to wave 1) and Menu. Dev mode adds Copy JSON.
- **PAUSED:** unchanged, including auto-pause on `visibilitychange`. Pausing during WAVE_CLEAR is a no-op (it's sub-600 ms and the bank is already frozen); if the tab hides during WAVE_CLEAR, complete the transition into PAUSED with the new grid built.

## 4. Seeds and determinism

- Per run: `runSeed = Date.now() % 2**31` (or dev-pinned via `?seed=`).
- Per wave: `waveSeed = mulberry32-mix(runSeed, n)` — e.g. seed the RNG with `(runSeed ^ (n * 0x9E3779B9)) >>> 0`. Layout of every wave derives entirely from `waveSeed`.
- A logged `(runSeed)` therefore reproduces the **entire run**, wave by wave. `attemptCounter` is deleted; `runCounter` (session-scoped, in-memory) numbers runs for telemetry.

## 5. Save schema v2

```js
// localStorage 'gridrush.v0' — bump schema
{ bestWave: 0, runsPlayed: 0, settings: { sound: true }, schema: 2 }
```

Migration: on load, `schema: 1` → carry `settings.sound`, initialize `bestWave: 0, runsPlayed: 0`, discard `highestUnlocked`, persist as schema 2. Corrupt/missing → defaults, silently. `bestWave`/`runsPlayed` update in `RUN_OVER` before the overlay renders.

## 6. Telemetry (`?dev=1`) — run-shaped

One record **per run**, not per wave:

```js
{ id, runSeed, outcome: 'red_tap' | 'timeout',
  waveReached, durationMs,
  waves: [{ n, clearMs, taps, wrongTaps, neutralTaps, bankAtClearMs }],
  bankAtDeathMs,               // ≤ 0 for timeout; snapshot for red_tap
  nextAction: 'retry' | 'quit' | 'abandoned' }
```

`bankAtClearMs` per wave is the tuning signal for §1's economy invariant: if median bank is still rising at wave 8, the economy is too generous — steepen `refundMs` decay; if median runs die before wave 4, soften it. Dev panel unchanged otherwise.

## 7. Acceptance criteria (delta)

1. Two consecutive runs both start at wave 1; nothing about run 1 (except `bestWave`/telemetry) affects run 2.
2. The bank visibly increases on a correct tap and on wave clear, never exceeds `capMs`, and is bit-frozen during WAVE_CLEAR (verifiable: background-tab-resume during a transition loses zero bank).
3. A red tap ends the run instantly regardless of bank.
4. Wave-clear-to-playing ≤ 600 ms; run-over-to-new-run ≤ 1 s, one tap.
5. Same `runSeed` reproduces identical layouts for every wave of the run.
6. `waveConfig(n)` for n = 1..50: neutrals ≥ 2, tiles cap at 6×6, all tiles ≥ 48 px on a 360 px-wide viewport.
7. Schema-1 saves migrate without data loss of the sound setting; `bestWave` persists across browser restarts (HTTP origin).
8. 10 rapid run restarts do not accelerate the timer (double-loop regression check, carried over).
9. `scripts/check.sh` passes; still one dependency-free `index.html`.

## 8. Out of scope for v1 (unchanged discipline)

Scoring beyond wave count · combo multipliers on refunds · shuffle/disguise/decoy mechanics (they slot in cleanly later as `waveConfig` flags) · leaderboards · story cutscene · PWA · haptics · monetization.

## 9. The v1 exit question

> Do playtesters' `waveReached` distributions **spread with practice** (runs 1→10 trending deeper), and do they keep pressing "Run again"? A flat distribution means the economy, not the player, decides run length — retune §1 before adding anything else.
